DPU implementation tutorial

This tutorial will guide you through the process of the new data processing unit (DPU) creation. But before you start coding there are few thing which deserve your attention.

Types of DPUs

The application distinguished three types of DPUs:
  • Extractor (DpuType.EXTRACTOR) - DPU which ...
  • Transformer (DpuType.TRANSFORMER) - DPU which ...
  • Loader (DpuType.LOADER) - DPU which ...

Logging the DPU's activity

Every DPU should log extensively. To log in your DPU, use slf4j. The log for each DPU is stored and accessible to the user by the graphical user interface of the application. Using sl4j is simple. If you are not familiar with sl4j the following example show the basics.
 
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class DPU_Template implements GraphicalExtractor {
	private Logger = LoggerFactory.getLogger(DPU_Template.class);
...
Then when you want to log something just call:
logger.debug("Some message .. ");

Configuration concept

Each DPU has possibility to store configuration that can configure DPU execution. User can edit the configuration through the configuration dialog which is also provided by DPU implementation. You can store any serializable object into configuration.

DataUnit containers

Data between DPUs are passed in containers that are called DataUnit. The DataUnit from the programmer's point of view is an interface that does not offer much functionality. For practical use more specific interface is provided. Currently only one interface is available: RDFDataRepository. The DataUnit (ie. RDFDataRepository) can't be created directly. To create DataUnit use DataUnitFactory, which is accessible via Context.

Context

Each DPU has execution context. It's main interface is ProcessingContext located in commons. The DPU itself (extractor/transformer/loader) uses more special type of Context ie. ExtractContext, TransformContext or LoadContext. The context provide ability to send messages that are stored and accessible to the user.
Context also manages access to the input/output DataUnits. The following example demonstrate this functionality for transformer: 

// get input DataUnits
List<DataUnit>inputs = context.getInputs();
// check if we have some inputs
if (inputs.isEmpty()) {
	throw new TransformException("Missing inputs!");
}
// use first input and ignore the rest
DataUnit dataUnit = inputs.get(0);
// cast input dataUnit to inputRepository, to check if we can cast we use 'instanceof'
RDFDataRepository intputRepository = null;
if (dataUnit instanceof RDFDataRepository) {
	intputRepository = (RDFDataRepository) dataUnit;
} else {
	throw new TransformException("Wrong input type " + dataUnit.getType().toString() + " expected RDF.");
}
// now, create output repository
// we have to specify type, in this case DataUnitType.RDF
RDFDataRepository outputRepository = (RDFDataRepository) context.getDataUnitFactory().create(DataUnitType.RDF);
// now we just check that we the object has been successfully created
if (outputRepository == null) {
	throw new TransformException("DataUnitFactory returned null.");
}

DPU implementation

It the following sections we will go through the files prepared in the DPU template project, from which you may derive your DPU. For each file we explain its purpose and list the mandatory and optionally changes that need to be done in order to implement DPU.
For each DPU, you have to:
  • adjust pom.xml containing basic metadata about the DPU
  • define the configuration dialog, which may be used by users of your DPU to set up the parameters of your DPU
  • implement the functionality of the DPU

pom.xml

The pom.xml file contains Maven configuration. If you are not familiar with Maven then just ignore everything except of several lines containing "TODO". TODOs have assigned number to make it easy to refer to them. In next few lines we go through all of them and give brief comment to each of them.
  • TODO 1. module_name: The Maven's artifact should be provided here. Maven uses this name to identify the output jar file. For common usage you can just use name of your DPU. This value should contains only a-z, A-Z, 0-9 and _ as whitespace. This value also determines the name of the module's main class.
  • TODO 2. DPU name: This value will be stored in manifest.mf file. This name is presented to the user when your DPU is imported into the application.
  • TODO 3. DPU version: In most common scenarios you can leave the default value here. This value is stored in manifest.mf file as "version of DPU". It correspond with Bundle-Version in manifest.mf.
  • TODO 4. DPU description: DPU's description is visible to the users. Provide all necessary information about your intended DPU usage here. Provide contact details on the author of the DPU.

DPU_Template.java

The DPU's main class. The name of the class and file must be the same as module_name. Thus, rename the file and class in it according to module_name. One important thing is that the package for this class must be module. As in pom.xml the template file contains "TODOs", places which requires your attention:
  • TODO 1. First of all we need to change the implemented interface of the DPU. The following interfaces are available:
    • GraphicalExtractor - Use if you implement new Extractor
    • GraphicalTransformer - Use if you implement new Transformer
    • GraphicalLoader - Use if you implement new Loader
    Each of these interfaces contains unique method that has to be implemented. We return to this later.
  • TODO 2. Method getType returns type of the DPU. So based on implemented interface return one of the following values:
    • DpuType.EXTRACTOR
    • DpuType.TRANSFORMER
    • DpuType.LOADER
    So just change the DpuType.EXTRACTOR value to one of those according to the implemented interface.
  • TODO 3. Here you should store default configuration of your DPU into configuration (the function parameter). For more information about configuration please refer to Configuration concept section.
    The examples how to store integer and string to database are presented in the function.
  • TODO 4. Here you should load the current DPU configuration from the given variable. But not to make it easy, if configuration dialog is open then you should load the configuration to the configuration dialog as well. If the configuration is invalid you should throw ConfigurationException. You can use the default implementation here.
  • TODO 5. Here you should save the current DPU configuration into given variable. Also you should save the configuration from the configuration dialog if it's open. If the configuration dialog contains invalid configuration you should throw ConfigurationException. You can leave the default implementation here.
  • TODO 6. Every DPU can have configuration dialog, that enable user to configure the DPU. In this method the DPU should create and return instance of this configuration dialog. The configuration dialog can exist as "the singleton". For common usage you can use the default implementation here.
  • TODO 7. Based on implemented interface (selected in TODO 1.) you have to provide implementation of one corresponding method required by that interface:
    • public extract(ExtractContext context) throws ExtractException for GraphicalExtractor
    • public void transform(TransformContext context) throws TransformException for GraphicalTransformer
    • public void load(LoadContext context) throws LoadException for GraphicalLoader
    In this method the functionality of the DPU should be implemented. All important data needed is passed to the method as Contex. To get more information to the Context please refer to the Context section above.

ConfigDialog.java

This file contains implementation of DPU's configuration dialog. The DPU's configuration dialog is implemented as a Vaadin component. To preserve compatibility with DPU_Template.java default's implementation the methods getConfiguration and setConfiguration must be preserved.
